# 带说话人 TXT -> 标准发言记录规范

本规范用于指导 agent 或人工流程，把一个已经带说话人标注的 TXT 文件整理为后续导入公开归档房间可直接使用的标准发言记录。

适用输入示例：

- `data/新闻价值与人伦道德谁更重要/test.txt`
- `data/钱是不是万恶之源/test.txt`

## 目标

- 根据语义与上下文，把原始发言归并为三类：`A`、`B`、`other`
- `A` 固定表示正方
- `B` 固定表示反方
- `other` 表示主持人、评委、解说、赛果播报、串场等非正反双方发言
- 同一个最终发言人相邻说出的内容必须合并成一个 turn，不要碎片化
- 最终产物要尽量精简，避免保留无用冗余信息，减少后续 token 消耗

## 输入假设

- 输入是带说话人标签的纯文本，常见格式如下：

```text
[SPEAKER_05]: 那么接下来我们就要引入今天的辩题了
[SPEAKER_08]: 众所周知记者也许是当今中国最尴尬的一个行业
```

- 原始 `SPEAKER_XX` 只是转录工具给出的标签，不保证真实身份正确
- 同一人可能被拆成多个 `SPEAKER_XX`
- 不同人也可能被错误合并
- 文本可能存在 ASR 错字、断句不完整、繁简混杂、角色词误识别等问题

## 输出格式

标准发言记录使用 JSON，对后续导入来说只保留必要字段：

```json
{
  "schemaVersion": "debate-record/v1",
  "title": "新闻价值与人伦道德谁更重要",
  "turns": [
    {
      "side": "other",
      "speaker": "主持人",
      "content": "那么接下来我们就要引入今天的辩题了。"
    },
    {
      "side": "A",
      "speaker": "正方",
      "content": "众所周知记者也许是当今中国最尴尬的一个行业……"
    },
    {
      "side": "B",
      "speaker": "反方",
      "content": "谢谢主席。对方立论今天刻意混淆了新闻价值和新闻道德两个概念……"
    }
  ]
}
```

仅在确实存在明显不确定性时，才允许额外附带：

```json
{
  "quality": {
    "needsReview": true,
    "notes": [
      "SPEAKER_05 与 SPEAKER_07 可能都是主持串场"
    ]
  }
}
```

## 字段定义

- `schemaVersion`
  - 固定为 `debate-record/v1`
- `title`
  - 辩题标题
  - 应该是稳定、简短、可作为房间标题直接使用的字符串
- `turns`
  - 按最终展示顺序排列的发言数组
- `turns[].side`
  - 只能是 `A`、`B`、`other`
- `turns[].speaker`
  - 当 `side = A` 时固定写 `正方`
  - 当 `side = B` 时固定写 `反方`
  - 当 `side = other` 时写具体角色，优先使用 `主持人`、`评委`、`解说`、`旁白`、`赛果播报` 等明确称呼；无法判断时写 `其它`
- `turns[].content`
  - 最终保留下来的发言正文
  - 必须是单段字符串，不拆成多段

## 归类规则

### 1. side 归类优先级

优先保证 `A/B/other` 正确，其次才是更细的角色名。

- 能明确判断属于正方，则标成 `A`
- 能明确判断属于反方，则标成 `B`
- 无法归到正反双方，或明显属于主持/评委/解说/广告串场，则标成 `other`

### 2. 正反方发言人的表示

- 不再细分一辩、二辩、三辩、四辩
- 只要最终属于正方，`speaker` 一律写 `正方`
- 只要最终属于反方，`speaker` 一律写 `反方`

这样做是为了和后续房间中的辩手 A / 辩手 B 展示保持一致，同时避免无意义地保留过细粒度身份信息。

### 3. other 的表示

- `other` 需要尽量保留角色信息
- 常见推荐写法：
  - `主持人`
  - `评委`
  - `解说`
  - `旁白`
  - `赛果播报`
  - `其它`

## 身份识别启发式

### 高优先级线索

- 主持引导语：
  - `有请正方一辩`
  - `有请反方三辩`
  - `请正方代表`
  - `接下来有请反方`
  - `现在进入自由辩论`
  - `有请评委点评`
- 阵营词：
  - `正方`
  - `反方`
  - `我方`
  - `对方`
- 角色词：
  - `主席`
  - `主持`
  - `评委`
  - `老师`
  - `观众`

### 模糊识别要求

必须容忍转录误差，例如：

- `一辩 / 一边 / 一变 / 一遍`
- `正方 / 政方 / 郑方`
- `反方 / 范方`

不要因为角色词有 ASR 错字就放弃识别。

### 原始 SPEAKER 标签处理

- `SPEAKER_XX` 不能直接当最终身份
- 如果多个 `SPEAKER_XX` 从语义上明显是同一主持人或同一阵营，应当归并
- 如果无法确认是不是同一个具体人，但可以确认同属正方或反方，则直接归到对应阵营即可，不必强行保留多个正方/反方身份

## 内容清洗规则

允许：

- 去掉开头的说话人标签，例如 `[SPEAKER_05]:`
- 合并被错误切开的同一连续发言
- 收敛多余空白字符
- 修正常见重复拼接，例如前后句明显重叠
- 对明显的 ASR 错字做保守纠错，例如角色词、常见连接词、非常确定的口误
- 在不改变原意的前提下补最基本的标点，便于后续阅读与导入展示
- 对明显的残句、重复尾巴、孤立碎词做最小修复，例如 `又有抢劫案又...`、`请问...很简` 这类被切断的片段

不允许：

- 改写论点
- 润色语气
- 补写原文没有表达的内容
- 把多个人的交锋总结成自己的话
- 把整段口语发言大幅改写成全新的书面表达
- 对不确定的专有名词、照片名、人名、节目名进行自信硬猜
- 明知内容已经断裂却原样输出一个明显不完整的句子

### 专有名词处理

- 如果专有名词高度确定，可以纠正为常见正确写法
- 如果只有大致猜测，优先保留原有近似写法，或者保留较中性的说法
- 如果该不确定性会影响理解，可在 `quality.notes` 中说明

### 残缺片段处理

- 如果一句话明显被切断，但可以根据上下文低风险补全连接关系，可以做最小修复
- 如果一句话尾部明显断裂、重复、缺主干，而你又无法低风险修复，优先删除那一小段不确定尾巴，不要把残缺噪声直接留进最终 `content`
- 如果删除或修复了这类残缺片段，且你认为会影响严格追溯，可在 `quality.notes` 中说明

## 合并规则

### 必须合并的情况

- 最终 `side` 相同
- 最终 `speaker` 相同
- 中间没有别人插话

这时应合并为一个 turn。

### 不应合并的情况

- 虽然同属 `A`，但中间已经被 `B` 或 `other` 打断
- 虽然都属于 `other`，但角色不同，例如主持人与评委不能合并

## 建议剔除内容

默认建议剔除纯广告或节目口播，除非它对理解辩论流程有直接价值，例如：

- 纯冠名广告
- 与辩论无关的节目包装语

建议保留：

- 主持开场与流程推进
- 评委提问
- 评委点评
- 比赛结果播报

## 输出质量要求

- `turns` 顺序必须与最终事件顺序一致
- 每个 `content` 都必须是非空字符串
- 不要保留原始 `rawLabels`、原始逐行索引、冗余说明等高 token 成本字段，除非另行生成独立审计文件
- 如果判断明显不稳，使用可选的 `quality` 字段标记，而不是偷偷猜测
- 如果已经做了人工复核且没有明显疑点，可以省略 `quality`

## 输出前自检

输出前建议自检以下问题：

1. JSON 是否能直接被解析
2. `schemaVersion` 是否固定为 `debate-record/v1`
3. `A` 的 `speaker` 是否全部是 `正方`
4. `B` 的 `speaker` 是否全部是 `反方`
5. 是否错误保留了广告口播、节目包装语等无关内容
6. 是否把相邻且同角色的碎片发言合并干净
7. 是否存在明显过度润色或“替原文重写”的情况
8. 是否仍然残留明显断裂句、重复尾巴、孤立碎词或 `...` 式占位

## 推荐工作流程

1. 读取整个 TXT，不要只看片段
2. 先识别主持/评委/解说等 `other`
3. 再根据流程锚点识别 `A/B`
4. 把连续的同角色发言合并
5. 输出为 `debate-record/v1` JSON
6. 只在必要时附带 `quality`
